# Project Adapt — 專案智慧適配

掃描當前 codebase，產生或審計 `.claude/project-profile.md`，讓模板化內容貼合專案實際狀況。

## 模式

- 不帶參數：預設 `init`（如果 profile 不存在）或 `audit`（如果已存在）
- 帶參數：`init` 或 `audit` 強制指定模式

---

## init — 首次產生

掃描 codebase → 產生完整 project-profile.md → 使用者審核確認。

**如果 `.claude/project-profile.md` 已存在，先警告並確認是否覆蓋。**

### Phase 1: 偵測專案技術棧

掃描以下檔案辨識語言、框架、build 工具：

| 偵測檔案 | 推斷技術 | 對應 Stack |
|----------|---------|-----------|
| `package.json` + `next.config.*` | Next.js | nextjs |
| `package.json` + `nuxt.config.*` | Nuxt | nuxt |
| `package.json` + `nest-cli.json` 或 `@nestjs/core` dep | NestJS | nestjs |
| `pubspec.yaml` | Flutter/Dart | flutter |
| `*.csproj` 或 `*.sln` | .NET | dotnet |
| `requirements.txt` 或 `pyproject.toml` + `fastapi` dep | Python FastAPI | python-fastapi |
| `composer.json` + `codeigniter4` dep | CodeIgniter 4 | ci4 |
| `prisma/schema.prisma` 或 `docker-compose.yml` (db service) | Database | database |
| `.prettierrc` / `.eslintrc` / `jest.config.*` / `vitest.config.*` | Testing/Linting | testing |
| `.pen` files | Pencil UI | pencil-ui |

多技術棧專案（如 monorepo）應列出所有偵測到的棧。

### Phase 2: 分析目錄結構

1. 列出前三層目錄，排除 `node_modules`/`.git`/`build`/`.dart_tool`/`dist`/`.next`/`vendor`/`__pycache__`
2. 歸納架構模式：Monorepo / Clean Architecture / MVC / Flat / Custom
3. 列出主要目錄與其用途（一句話描述）

### Phase 3: 偵測 Build & Test 指令

| 來源 | 抽取方式 |
|------|---------|
| `package.json` scripts | 讀取 `scripts` 物件，抽取 `build`/`test`/`dev`/`start`/`lint` |
| `Makefile` | 讀取主要 target 名稱 |
| `pubspec.yaml` | 推薦 `flutter build`/`flutter test` |
| `*.csproj` | 推薦 `dotnet build`/`dotnet test` |
| `pyproject.toml` | 讀取 `[tool.pytest]` 或 `[project.scripts]` |
| `composer.json` scripts | 讀取 `scripts` 物件 |

產出格式：`| Stack | Command | Purpose |` 表格。

### Phase 4: 偵測 Coding Style

掃描以下設定檔：

| 檔案 | 抽取內容 |
|------|---------|
| `.eslintrc*` / `eslint.config.*` | Linting 規則摘要 |
| `.prettierrc*` | 格式化設定（縮排、行寬、trailing comma 等） |
| `.editorconfig` | 縮排、編碼、換行 |
| `pyproject.toml` [tool.black/ruff] | Python 格式化設定 |
| `analysis_options.yaml` | Dart/Flutter lint 規則 |
| `tsconfig.json` strict 設定 | TypeScript 嚴格模式 |

若無設定檔，標記為「未偵測到，建議建立」。

### Phase 5: 推薦技術棧 Agents

1. 讀取 `.claude/manifest.json` 的 `agents` 陣列（目前已安裝的 agents）
2. 對比 Phase 1 偵測到的技術棧
3. 列出表格：

| Agent | Status | 說明 |
|-------|--------|------|
| {agent-name} | ✅ 已安裝 / ⚠️ 建議啟用 / ➖ 不相關 | {理由} |

只列「已安裝且相關」和「未安裝但建議啟用」，不列不相關的。

### Phase 6: 產生 Repo 特化守則

基於偵測到的架構模式與技術棧，產生 3-5 條守則。參考模式：

| 架構模式 | 守則範例 |
|----------|---------|
| NestJS | Controller 禁止直接操作 DB；Service 層處理業務邏輯 |
| Flutter | UI 元件禁止硬編 Colors；網路層使用統一 client |
| Next.js | API routes 統一放 `app/api/`；Server Components 優先 |
| FastAPI | Router 與 service 分離；Pydantic model 做驗證 |
| Clean Architecture | 依賴方向由外向內；Entity 層禁止 import framework |
| Monorepo | 跨 package import 必須走 package.json 宣告 |

**守則必須基於掃描到的實際 codebase 結構產生，不可憑空杜撰。**
如果偵測到的程式碼太少無法推斷，該區段標記為「需手動補充」。

### Phase 7: 組合輸出

將 Phase 1-6 的結果組合成以下格式：

```markdown
# Project Profile — {project-name}

> Auto-generated by project-adapt skill. 手動新增的內容會被保留。
> Last generated: {date}

## Project Identity

- **Name**: {偵測到的專案名稱}
- **Primary Language**: {主要語言}
- **Framework**: {框架}
- **Architecture Pattern**: {架構模式}
- **Installed Stacks**: {已安裝的 ChillVibe stacks，逗號分隔}

## Project Structure

{Phase 2 產出}

## Build & Test Commands

| Stack | Command | Purpose |
|-------|---------|---------|
{Phase 3 產出}

## Coding Style

{Phase 4 產出}

## Tech Stack Agents

| Agent | Status | 說明 |
|-------|--------|------|
{Phase 5 產出}

## Repo-Specific Rules

{Phase 6 產出，numbered list}
```

### Phase 8: 使用者審核

1. 顯示產生的完整 profile 內容
2. 詢問：「以上 profile 是否正確？可修改後確認，或直接確認寫入。」
3. 使用者確認後寫入 `.claude/project-profile.md`
4. 提示：「Profile 已寫入。下次 fleet-update 不會覆蓋此檔案。」

---

## audit — 檢查覆蓋度

重新掃描 codebase，與現有 `project-profile.md` 比對，**只產出報告不修改檔案**。

### 執行步驟

1. 讀取現有 `.claude/project-profile.md`（不存在則提示先跑 init）
2. 執行 Phase 1-6 重新掃描
3. 逐區段比對，產出報告：

```
=== Project Adapt Audit Report ===

Project Identity:  CURRENT ✅
Project Structure: DRIFT ⚠️ (新增了 src/middleware/ 目錄)
Build & Test:      CURRENT ✅
Coding Style:      CURRENT ✅
Tech Stack Agents: MISSING ⚠️ (偵測到 Prisma 但未推薦 prisma-engineer)
Repo Rules:        CURRENT ✅

Overall: 4/6 CURRENT, 1 DRIFT, 1 MISSING
建議：執行 $skill-project-adapt init 重新產生 profile
```

狀態定義：
- `CURRENT`：profile 與 codebase 一致
- `DRIFT`：profile 內容與實際不符（列出具體差異）
- `MISSING`：codebase 有新技術/目錄但 profile 未涵蓋
