# Spec Review 審查標準

## 審查範圍規則

> 核心原則：審查焦點為 **S1 dev spec（`s1_dev_spec.md`）**。
> S0 brief spec（`s0_brief_spec.md`）由用戶人工確認需求正確性，審查時：

- 讀取 S0 作為背景上下文
- 不對 S0 內容提出 P0/P1 修改要求
- 若 S1 與 S0 有矛盾，以建議形式附在報告末尾（非阻斷）
- 不質疑 S0 中的需求定義本身

## 專案架構

> 依據 repo 的 `CLAUDE.md` / `AGENTS.md` 中 Repository Guidelines 所定義的專案結構與技術棧。

## 審查項目

### 1. S1 Dev Spec 完整性

- 任務清單是否有可驗證 DoD
- 驗收標準是否可測試
- User Flow 是否涵蓋主要/異常/邊界情境
- Data Flow 是否完整（從 UI 到 API 到資料層）
- 影響範圍是否完整
- 風險評估是否識別關鍵風險
- 技術決策是否有充分依據

### 2. 技術合規

- 是否遵循 repo 規定的架構模式與分層
- 是否使用 repo 規定的 API 呼叫方式
- UI 是否使用專案設計系統元件
- 狀態管理是否符合專案慣例
- 資料庫安全（不刪資料）是否被遵守

### 3. 與 codebase 一致性

- Spec 提到的 class/method/endpoint 是否與現有程式碼吻合
- DTO 欄位是否與 Entity 定義一致
- 路由命名是否符合既有慣例

## 問題分類

| 等級 | 類型 | 說明 |
|------|------|------|
| P0 | 必須修正 | Spec 錯誤、重大遺漏、架構衝突 |
| P1 | 需修正 | 邏輯缺口、不完整實作規格 |
| P2 | 建議 | 可優化但不阻擋 |
